# Выбор задачи: Out-of-Scope Detection

**Дата:** 2025-03
**Спринт:** 1

## Почему OOS Detection, а не Jailbreak Detection

1. **Прямая связь с AutoIntent.** AutoIntent разрабатывался
   под intent detection + OOS как нативный класс. Интеграция
   чистая, без архитектурных сюрпризов.

2. **Чистый датасет.** CLINC150 — стандартный бенчмарк с
   тремя уровнями сложности OOS, скачивается за минуту,
   результаты воспроизводимы.

3. **Референсные числа из статьи.** AutoIntent (EMNLP 2025)
   уже публикует результаты на CLINC150 (OOS F1 = 76.79).
   Это даёт точку проверки воспроизводимости.

4. **Риск потерять спринт — минимальный.** Jailbreak Detection
   требует больше времени на подбор датасета и постановку.

## Почему это отличается от постановки в статье AutoIntent

Статья сравнивает AutoIntent с другими AutoML-фреймворками
(AutoGluon, H2O) — вопрос "наш AutoML лучше других AutoML".

Наша постановка: **"применим ли AutoIntent как guardrail в
production?"** Ключевые отличия:
- Метрики под guardrail-контекст (Recall@FPR вместо accuracy)
- Few-shot режим как основной сценарий
- Latency как дополнительное измерение
- SOTA-бейзлайн — специализированный OOS-метод, а не другой AutoML
