# AutoGuardrails

AutoML-фреймворк для построения защитного периметра (guardrails) LLM- и агентных приложений. Проект проверяет применимость фреймворка **AutoIntent** (AutoML для классификации текста) к задачам безопасности и дорабатывает его там, где базового качества недостаточно.

**Конечная цель:** система, которая по описанию приложения (система, промпты, тип риска) автоматически подбирает и обучает guardrail-модули из небольшого набора примеров.

> **Статус.** Активная разработка. Вся работа ведётся в двух ветках задач — `oos_detection` и `jailbreak_detection`. В `main` они **не смержены**: по обеим задачам продолжаются доработки и допрогоны. Результаты предварительные.

## Задачи

Две репрезентативные guardrail-задачи как proof-of-concept: проверка переноса AutoML на guardrails и границ его применимости.

**Out-of-Scope Detection** — детекция запросов вне области компетенции системы. Датасет CLINC150 (150 интентов + OOS-класс) в двух конфигурациях с идентичным тест-сплитом (`standard` — 100 OOS в train, `deeppavlov` — 200). Метрики: OOS Recall, in-domain accuracy, F1 OOS.

**Jailbreak Detection** — детекция атак, обходящих защиту модели. Бинарная классификация (safe / jailbreak) на WildJailbreak (NeurIPS 2024). Ключевая бизнес-метрика — **Over-refusal Rate (ORR)**, доля ложно заблокированных безопасных запросов.

Сравнение по двум осям: универсальные AutoML-фреймворки (AutoGluon, LightAutoML, H2O) и доменные SOTA-модели (PromptGuard 2, Qwen3Guard).

## Ключевые результаты

**OOS Detection** (source `standard`, few-shot, на единых e5-эмбеддингах, предварительно). AutoIntent лидирует по F1 OOS на всех few-shot уровнях; LightAutoML даёт выше in-domain accuracy, но уступает по OOS. Преимущество AutoIntent — эффективность обучения (~110 с против сотен–тысяч секунд у фреймворков). Расхождение с Table 3 статьи AutoIntent — в процессе выяснения (конфигурация датасета и эмбеддер как причины исключены).

**Jailbreak Detection.** Последовательная диагностика (ноутбуки 06–15) исчерпала возможности scoring- и decision-узлов и локализовала узкое место на уровне **представления**:
- на общих e5-эмбеддингах все модели упираются в близкий потолок ранжирования (ROC-AUC ~0.76–0.82);
- настроенный LightGBM (0.819) догоняет лучшие AutoML, но не пробивает потолок; отрыв статистически незначим; CatBoost слабее (0.800); стекинг прироста не даёт;
- high ORR (лучший достижимый ~0.36) — следствие потолка разделимости, а не выбора порога;
- препроцессинг бесполезен: векторы атак в данных распределены симметрично между safe/harmful;
- вывод: дальнейший прирост возможен только на уровне эмбеддинга.

## Дальнейшие планы

- **OOS:** допрогоны на нескольких сидах, вариант `--decision-metric oos_f1`, переизмерение latency, перепрогон H2O на x86/Windows.
- **Jailbreak:** прогон доменных SOTA на наших подвыборках.
- **Доработка фреймворка:** главное направление — embedding-узел (instruction-префикс, склейка со скрытым состоянием safety-классификатора, более сильные эмбеддинги); как готовое улучшение — LightGBM-scorer в пресет `classic`.
- **Развитие:** третья категория риска (prompt injection); прогон AutoML-фреймворков в дефолтном режиме обработки текста.
